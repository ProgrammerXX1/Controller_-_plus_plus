# tests

Стенд для замеров производительности C++ на изолированных ядрах.

## Структура

```
tests/
├── README.md
├── quicksort_levels.cpp   # пять реализаций quicksort на разных уровнях абстракции
└── data/
    └── hn_1m.jsonl        # 1 000 000 записей Hacker News, 310 MB
```

## Аппаратное окружение

- **CPU:** Intel i9-11900, 8 физ. ядер × 2 HT = 16 логических, single NUMA, 16 MiB L3
- **Изолированные ядра:** `0, 1, 8, 9` — две полные физические пары (CPU0↔CPU8, CPU1↔CPU9)
- **Boot params:** `isolcpus=0,1,8,9 nohz_full=0,1,8,9 rcu_nocbs=0,1,8,9`
- На «общих» ядрах остаётся 12 логических (2-7, 10-15) — на них живёт всё остальное в системе

Изоляция позволяет получать стабильные цифры: на ядрах 0,1,8,9 не крутится ни планировщик пользовательских задач, ни RCU-коллбэки, ни (для CPU 1,8,9) тик планировщика.

### Как запускать бенчмарк на изолированных ядрах

```bash
# простой пин на конкретные CPU
taskset -c 0,1,8,9 ./qs

# с RT-приоритетом — вытесняет даже kworker
sudo chrt -f 50 taskset -c 0,1 ./qs
```

Без `taskset` процесс унаследует маску шелла (`2-7,10-15`) и на изолированные ядра не попадёт.

## quicksort_levels.cpp

Пять реализаций сортировки от STL до ручной оптимизации — для сравнения производительности и иллюстрации идиом современного C++.

| Уровень | Что показывает |
|---|---|
| 1. `std::sort` | эталон: introsort + insertion sort |
| 2. Идиоматичный | шаблоны + итераторы + value semantics |
| 3. Сырые указатели | C-style, Lomuto partition |
| 4. Итеративный | свой стек на арене, без рекурсии |
| 5. Оптимизированный | median-of-three + Hoare + insertion sort на малых, tail-call elim |

### Сборка и запуск

```bash
g++ -O3 -std=c++20 -o qs quicksort_levels.cpp
taskset -c 0,1,8,9 ./qs
```

Бенчмарк: 5 000 000 случайных `int`, best-of-5 каждой реализации, проверка `std::is_sorted` после каждой итерации.

## data/hn_1m.jsonl

1 000 000 записей Hacker News (story-level), формат:

```json
{"id": "...", "title": "...", "author": "...", "url": "...",
 "text": "...", "points": N, "num_comments": N, "created_at_i": <unix_ts>}
```

| Параметр | Значение |
|---|---|
| Размер | 310 MB |
| Записей | ровно 1 000 000, валидный JSON в каждой строке |
| Покрытие | ~18 лет (2008-03 — 2026-05) |
| Длина `text` | median = 0 (link-posts), mean = 136 B, p95 = 1 KB, max = ~9 KB |

Это **один файл с миллионом строк**, а не миллион файлов на диске — намеренно: один sequential read вместо миллиона `open()`, легко мапится через `mmap`, прогревается в page cache после первого прогона.

Применения: индексация и поиск, BM25/TF-IDF, JSON-parse бенчмарки (simdjson / RapidJSON / nlohmann), сортировка по `points` или `created_at_i`, hash-table benchmarks (group by author).
